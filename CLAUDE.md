# PRODUCT DELIVERY ENGINEER — EXECUTION WORKFLOW

> Эта секция описывает **КАК** выполняется каждый запрос. Она дополняет, а не заменяет
> правила управления выше (`TASK MANAGEMENT`, `GITHUB WORKFLOW`, `TESTING FIRST`,
> `DEFINITION OF DONE`). При конфликте формулировок — приоритет у более строгого требования.

## Mission

Цель — не написать код, а поставить работающую функциональность, которая переживает
тестирование, валидацию и реальную эксплуатацию.

Задача завершена только когда (в дополнение к `DEFINITION OF DONE`):

* реализация закончена;
* тесты проходят;
* регрессии устранены;
* затронутые пользовательские сценарии работают;
* функция интегрирована с существующей архитектурой.

## Operating principle: inspect before asking

Не задавать вопросы, ответ на которые есть в: исходном коде; структуре репозитория;
истории коммитов; документации; текущей реализации.

При неполных требованиях: исследовать существующую архитектуру → существующие паттерны →
сопоставимые реализации → выбрать наиболее консервативное решение, согласованное с
архитектурой. **Consistency over novelty.** Не изобретать новую парадигму, если в проекте
уже есть устоявшийся паттерн.

## Per-request pipeline

Для каждого запроса выполнять конвейер:

**Phase 1 — Repository Inspection.** Изучить: архитектуру, модули, маршруты, state
management, слой данных, API-интеграции, права доступа, навигацию, существующие тесты,
дизайн-систему. Определить затронутые области. Не трогать несвязанные модули.

**Phase 2 — Functional Analysis.** Определить: ожидаемое / текущее / отсутствующее /
сломанное / неполное поведение. Составить карту зависимостей, побочных эффектов и
затронутых пользовательских сценариев.

**Phase 3 — Implementation.** Реализовать изменение. Требования: архитектурная
согласованность; без дублирования; без временного кода; без TODO-заглушек; без mock-ов,
если они не запрошены явно. Только production-качество.

**Phase 4 — Regression Prevention.** До завершения проверить затронутые сценарии:
навигацию, переходы состояний, persistence данных, права доступа, loading/error/empty
states, граничные случаи. Найденные регрессии исправлять немедленно.

**Phase 5 — Testing & Deployment Verification.** Запустить релевантные тесты в порядке:
unit → integration → e2e (если применимо) → build → production build. При падении: найти
первопричину → исправить → перезапустить затронутые и зависимые тесты. **Никогда не
останавливаться на первой ошибке. Никогда не отмечать работу завершённой при падающих
тестах.** Для изменений, видимых в приложении, дополнительно проверять на **развёрнутом
окружении** (см. `Lessons enforced`).

## Product Quality Standard — эталоны (НЕ для копирования)

Использовать зрелые системы (Jira, Linear, ClickUp, Asana, Trello, Monday) как
**референсы качества**, а не как объект клонирования. Сверять по ним: качество workflow,
UX-паттерны, информационную архитектуру, поведение управления задачами / спринтами /
backlog. Если текущая реализация объективно слабее проверенной индустриальной практики —
улучшить её. Не копировать интерфейсы и не дублировать фичи бездумно.

## Task management domain requirements (специфика WorkHelper)

* **Backlog** — отдельная логическая сущность, отдельный этап workflow и отдельный раздел
  интерфейса. Элементы backlog не отображаются в рабочих колонках доски до переноса в
  активный спринт.
* **Sprint Board** — только активные задачи спринта; колонки по статусам; корректные
  переходы между статусами; никаких backlog-элементов внутри доски.
* **Calendar** — привязка задач к датам, видимость дедлайнов, переход из календаря к задаче.
* **Notifications** — напоминания о дедлайнах и событиях, связь уведомлений с задачами.
* **Settings** — централизованная конфигурация с явной, обнаруживаемой точкой входа.

## UX validation

Проверять: desktop и mobile, адаптивность, empty / loading / error states, доступность,
визуальную консистентность. Удалять: дублирующие контролы, мёртвый/вводящий в заблуждение
UI, визуальный шум.

## Feature completion rules — фича НЕ завершена, если присутствует хотя бы одно:

сломанная навигация; недоступная функциональность; мёртвые кнопки; разорванные сценарии;
несогласованные состояния; необработанные ошибки; отсутствующая валидация; отсутствующий
persistence; падающие тесты.

## Completion Gate (расширяет `DEFINITION OF DONE`)

Работа завершена только если **все** условия истинны: реализация завершена; архитектура
остаётся когерентной; нет известных регрессий; тесты проходят; build успешен; production
build успешен; затронутые сценарии проверены; поведение корректно с точки зрения
пользователя; для разворачиваемых изменений — проверено на развёрнутом окружении.
Если хотя бы одно условие не выполнено — продолжать работу.

**Skeleton-first (ТП-185).** Ни одна новая модалка/панель не уходит в Done без
skeleton-first открытия (мгновенный каркас + известные данные, не блокирующий спиннер),
прогрессивной загрузки тяжёлых блоков, честных состояний loading/error/empty по блокам и
перф-проверки. Стандарт и примитивы — `.ai/SKELETON_FIRST_STANDARD.md`; эталон —
`TaskCardModal` + `TaskCardSkeleton`.

## Lessons enforced (проверенные уроки этого проекта)

Эти правила выведены из реальных дефектов, найденных в ходе работы над WorkHelper, и
обязательны к применению:

* **Deployed-environment verification.** Для изменений, видимых в приложении, проверять на
  реально развёрнутом окружении и предоставлять рабочую ссылку — а не только локальную
  сборку. «Собралось локально» ≠ «работает у пользователя».
* **Branch / merge integrity.** Перед утверждением, что фича есть или отсутствует, сверять
  ветки (`git log --all --oneline`, сравнение с `origin/*`, проникающие коммиты). Готовая
  функциональность может быть просто не смержена — это уже приводило к «потере» 8 коммитов.
* **Frontend ↔ Backend contract.** Каждый endpoint, который вызывает фронтенд, обязан
  существовать на бэкенде с совпадающим путём, методом и формой DTO. Проверять полный путь
  запроса end-to-end. Ответ 401/404 на вызов из UI = несуществующий или несогласованный
  контракт, а не «проблема авторизации».
* **Validators & permissions — частый источник блокеров.** Проверять граничные роли (только
  что зарегистрированный пользователь) и неактивные состояния сущностей (задача в backlog,
  архивный спринт). Бизнес-доступ решается в сервисе, валидатор проверяет только
  существование/формат.
* **Error hygiene.** Не возвращать внутренние детали (SQL, имена constraint, stacktrace) в
  ответах API — отдавать пользователю понятное сообщение, технические детали только в логи.
* **Secrets discipline.** Никогда не записывать реальные секреты в файлы или в текст команд;
  передавать через переменные окружения. Секреты не попадают в репозиторий (см. `SECURITY`).
* **Router-hooks вне роутера (ТП-172, прод-инцидент «белый экран»).** Компоненты,
  переиспользуемые NiceModal-модалками, НЕ используют `useNavigate`/`useRouter` —
  модалки живут вне Router-контекста и хук роняет всё дерево. Навигация из таких
  компонентов — только прямой `router.navigate` (`@/application/router`).
* **Error boundaries обязательны (ТП-172 T1).** Роуты покрыты `defaultErrorComponent`
  (ТП-131); любое дерево ВНЕ роутов (модалки) оборачивается
  `shared/ui/components/ErrorBoundary`. Краш блока ≠ белый экран приложения.
* **Post-mortem прод-багов (ТП-172 T7).** Сбежавший в прод баг сначала
  воспроизводится ПАДАЮЩИМ тестом, затем чинится; тест остаётся навсегда.
  Шаблон и реестр — `.ai/POSTMORTEM_TEMPLATE.md`. Гейт: незамеченный
  `console.error`/uncaught в тесте = красный тест (`src/test/setup.ts`).

## Backend (backend/)
- **Java 21**, Spring Boot 3.4.0
- Spring Security + JWT
- PostgreSQL + Liquibase
- Lombok, MapStruct
- Swagger / OpenAPI

### Запуск базы данных локально
```bash
cd backend/enviroment/local
docker-compose up -d
```

### Сборка, тесты и запуск
```bash
cd backend
./gradlew test       # юнит/интеграционные тесты
./gradlew build      # сборка + тесты
./gradlew bootRun    # запуск приложения
```

## Frontend (frontend/)
- **React 19**, TypeScript, Vite
- Material UI v7, TanStack Router/Query/Table
- Zustand, React Hook Form + Zod, Axios

### Установка, запуск и проверки
```bash
cd frontend
npm install
npm run dev      # дев-сервер (Vite)
npm run lint     # ESLint
npm run check    # prettier --write + eslint --fix
npm run build    # production build (tsc -b && vite build)
```

### Генерация типов из OpenAPI
```bash
npm run openapi-generate
```

> `build`). Если правишь его вручную — обнови все интерфейсы (`FileRoutesByFullPath`,
