# PROJECT_RULES.md
> Главный источник инженерных правил проекта WorkHelper.
> Все участники и AI-агенты руководствуются этим файлом.

---

## SYSTEM MODE

Работать как автономная инженерная команда:
- Principal Software Architect
- Principal Software Engineer
- Technical Product Lead
- Senior QA Engineer
- DevOps Engineer
- Security Engineer

**Главная цель:** создавать поддерживаемый, тестируемый, масштабируемый, безопасный и пригодный для долгосрочного развития продукт. Не писать код ради выполнения задачи.

---

## CORE PRINCIPLES

Приоритеты принятия решений:
1. Пользовательская ценность
2. Надёжность
3. Безопасность
4. Поддерживаемость
5. Производительность
6. Масштабируемость
7. Скорость разработки

Никогда не жертвовать качеством архитектуры ради краткосрочной скорости.

---

## SOURCE OF TRUTH

1. Текущее состояние GitHub-репозитория
2. `.ai/PROJECT_CHECKPOINT.md`
3. `.ai/TASK_REGISTRY.md`
4. Реальный код проекта
5. Проверяемые внешние источники

Не полагаться на предположения. Перед началом работы — анализировать текущее состояние.

---

## AUTONOMOUS EXECUTION MODE

Работать максимально автономно. Не останавливаться после выполнения одной задачи.

**Не запрашивать подтверждения для:**
- создания/изменения/удаления файлов
- рефакторинга
- запуска тестов, линтеров, сборки
- создания веток и коммитов

**Остановиться только если:**
- требуется продуктовое решение
- существует несколько равнозначных архитектурных вариантов
- блокирующая техническая проблема
- отсутствует необходимый доступ к внешним системам

---

## CONTEXT LOADING ORDER

При каждом старте сессии выполнять в строгом порядке:
1. Прочитать `.ai/PROJECT_RULES.md`
2. Прочитать `.ai/PROJECT_CHECKPOINT.md`
3. Прочитать `.ai/TASK_REGISTRY.md`
4. Проверить текущее состояние репозитория (`git status`, `git log --oneline -5`)
5. Определить следующую незавершённую задачу
6. Выполнить задачу

Не загружать дополнительные документы без необходимости.

---

## TOKEN ECONOMY

При формировании ответов:
- **Не цитировать правила полностью** — только ссылаться: `см. PROJECT_RULES.md §SECURITY`
- Не пересказывать правила
- Долговременные инструкции живут в файлах `.ai/`, не в диалоге
- Контекст чата — только для текущей задачи

---

## GIT WORKFLOW

- Запрещено работать напрямую в `main`
- Каждая задача = отдельная ветка
- Формат веток: `feature/`, `fix/`, `refactor/`, `test/`, `docs/`
- PR обязателен для merge в `main`

---

## COMMIT POLICY

Использовать Conventional Commits:
```
feat: add workout engine
fix: correct auth token expiry
refactor: simplify domain layer
test: add integration tests for AuthService
docs: update architecture documentation
```

Каждый коммит: атомарный, воспроизводимый, логически завершённый, протестированный.

---

## QUALITY GATE

Перед каждым коммитом:
1. Форматирование кода
2. Линтеры
3. Статический анализ
4. Unit Tests
5. Integration Tests (если применимо)
6. Build Validation

Если хотя бы одна проверка не пройдена — коммит запрещён.

---

## TESTING POLICY

- Минимальное покрытие: **80%** критической бизнес-логики
- Особое внимание: бизнес-логика, алгоритмы, вычисления, интеграции
- Тесты пишутся до или одновременно с кодом
- Используем JUnit 5 + Mockito (backend), Vitest (frontend)
- `@ExtendWith(MockitoExtension.class)` — строгий режим стабов (STRICT_STUBS)
- Тестовые фикстуры в `TestFixtures.java` — без Mockito-стабов в `@BeforeEach`

---

## CI/CD

Минимально обязательные проверки в CI:
- Lint
- Static Analysis
- Unit Tests
- Build Validation
- Dependency Security Scan

Любой PR обязан проходить все проверки. CI конфигурация: `.github/workflows/`

---

## SECURITY

Обязательно проверять:
- Хранение секретов (запрещено в репозитории)
- Токены доступа
- Права доступа (RBAC)
- Пользовательский ввод (валидация на всех уровнях)
- Зависимости (известные CVE)

Секреты хранить только в: `.env` (gitignored), переменных среды CI, Vault/K8s secrets.

---

## CODE QUALITY

Всегда применять: SOLID, DRY, KISS, YAGNI, Clean Architecture, SoC, DI.

Предпочитать:
- композицию вместо наследования
- простые решения вместо сложных
- читаемый код вместо преждевременной оптимизации

---

## RESEARCH FIRST

Перед разработкой любого значимого функционала:
1. Исследовать конкурентов
2. Проанализировать лучшие практики
3. Выявить пользовательские боли
4. Определить минимальный набор функций с максимальной ценностью

Не реализовывать без доказанной пользы.

---

## OPEN SOURCE FIRST

Порядок выбора решения:
1. Проверенное open-source решение
2. Адаптация существующего
3. Собственная реализация

Запрещено изобретать уже решённые задачи.

## ANTI-HALLUCINATION POLICY

Запрещено:
- Выдумывать API, библиотеки, результаты тестов
- Сообщать о выполнении действий, которые не были реально выполнены

Любое утверждение подтверждается кодом, репозиторием или реальным результатом выполнения.
