# VOICE_ARCHITECTURE.md — подсистема голосового управления WorkTask

> ADR и проектный документ подсистемы «Голосовой помощник» (ТП‑90 / V0).
> Дополняет `ARCHITECTURE.md` (раздел «Voice Control Subsystem», ADR‑005..010).
> Это архитектурная декомпозиция, а не жёсткая спецификация: каждая задача
> дерева проходит повторный инженерный анализ перед реализацией.
>
> **Статус:** утверждён пользователем 2026‑07‑04 (MVP = Веха 1 без AI; NLU =
> backend‑прокси с провайдером за интерфейсом; дерево заведено как ТП‑90..103).
> **Кода в этой задаче нет** — только документация и контракты.

---

## 1. Контекст и цель

Цель подсистемы — сделать голос **полноценным способом работы** с WorkTask, а не
дополнительной диктовкой. Пользователь разговаривает естественным языком; система сама
определяет намерение, контекст и последовательность действий, **используя только то, что
пользователь сообщил явно или что уже есть внутри WorkTask** (никаких выдуманных действий).

Конечные возможности (не все сразу, но архитектура обязана их допускать): создание задач и
багов, редактирование, комментарии, смена исполнителя/приоритета/статуса/спринта, создание и
завершение спринтов, поиск/фильтрация/навигация, встречи и календарь, проекты, уведомления —
т.е. **любое действие, доступное через обычный интерфейс**.

### Ключевое наблюдение
Пространство действий уже существует — это API‑поверхность `workTechApi.*`
(task/sprint/project/meeting/notification/user/status). Требуемые голосовые возможности
отображаются на неё практически 1:1. Поэтому голосовой помощник — это **новая модальность
ввода поверх той же командной поверхности, что и UI**, а не параллельная бизнес‑логика.

**Инвариант №0:** голос обязан выполнять действия через те же мутации/сервисы, что и UI
(`useCreateTask`, `useUpdateTaskStatus`, `updateTasksSprint` → `TaskPlacementService` и т.д.).
Прямых обращений в API/БД в обход существующего слоя быть не должно.

## 2. Прежние решения в WorkTask (prior art — переиспользуем)

Голос в проекте уже проходил две итерации. Обе — источник переиспользуемых решений:

### 2.1 ТП‑22 / ТП‑57 — командный микрофон (удалён в ТП‑88)
Существовал реестр команд (`features/voice/commandRegistry.ts`, `types.ts`,
`commands/createTaskCommand.ts`, `useVoiceHotkey.ts`, `VoiceControl.tsx`), удалённый в
коммите `211d152`. Восстановим из истории (`git show 211d152~1:...`).

Что было **спроектировано хорошо** (переиспользуем):
- `VoiceCommand` — дескриптор команды (`id`, `execute(payload, ctx)`) с самостоятельной
  регистрацией в реестре: «новая команда = новый файл + `registerVoiceCommand`, существующий
  код не меняется».
- `VoiceCommandContext` — сервисы, инъецируемые командам (`projectId`, `defaultSprintId`,
  `createTask`, `navigate`). Это ровно наш «Executor делегирует существующим мутациям».
- `VoiceCommandResult` (`message`, `taskCode`) — единый формат обратной связи.
- `useVoiceHotkey` — раскладко‑независимый хоткей через `e.code` (урок ТП‑57: на русской
  раскладке `e.key` ломает шорткат). Переиспользуем как точку входа лаунчера (X1).

**В чём был изъян** (из‑за него удалили): распознавание намерения было размазано по командам —
каждая команда сама regex‑матчила сырой транскрипт (`match(transcript)`), а `dispatchTranscript`
опрашивал их по очереди. На свободную речь это давало «Команда не распознана»: не было ни
единого слоя намерения, ни учёта контекста, ни извлечения слотов. Плюс глобальный микрофон в
шапке конфликтовал с моделью «голос = ввод по контексту».

### 2.2 ТП‑88 — голосовой ВВОД, этап 1 (актуальный код)
`features/voice/core`: конвейер `SpeechRecognition → IntentAnalyzer → TextFormatter → Executor`.
`IntentAnalyzer` — **намеренно оставленная точка расширения** («этап 2 заменит заглушку
LLM‑реализацией того же интерфейса без изменений в остальном коде»). Работает: диктовка в поля
и разбиение «название·описание» при создании. Локально (Web Speech, ru‑RU), без ключей.

### 2.3 Вывод для архитектуры
Мы **не изобретаем новый паттерн и не делаем ревёрт**. Мы соединяем два уже существующих в
проекте решения и устраняем изъян 2.1:
- берём реестр/дескриптор/контекст/результат из 2.1;
- **убираем распределённый `match()`** и заменяем его **центральным слоем распознавания
  намерения** (2.2, шов `IntentAnalyzer`), который выбирает команду и извлекает слоты;
- команда сохраняет только `execute(slots, ctx)`.

## 3. Общее решение

Слоистый конвейер (обобщение существующего из ТП‑88). Каждый слой — одна ответственность,
заменяем независимо (Strategy / DI).

```
┌──────────────────────────────────────────────────────────────────────┐
│  SpeechProvider     Web Speech API (ru-RU) → плагин (Whisper/Yandex)   │  есть (ТП-88)
│                     push-to-talk; interim + final транскрипт           │
├──────────────────────────────────────────────────────────────────────┤
│  VoiceContext       активный проект/спринт, открытая задача (route),   │  ТП-92 (F2)
│                     текущий пользователь, справочники (участники,       │
│                     статусы, спринты, приоритеты)                       │
├──────────────────────────────────────────────────────────────────────┤
│  IntentResolver     (шов IntentAnalyzer из ТП-88)                       │  ТП-94/96 (I1/I2)
│    ├─ RuleBasedResolver   грамматика правил, БЕЗ ключей (Веха 1)        │
│    └─ LlmResolver          backend-прокси, function-calling (Веха 2)   │
│    вход: transcript + context + схема команд из реестра                 │
│    выход: IntentResolution { commandId, slots, confidence,             │
│                              needsConfirmation, clarification? }         │
├──────────────────────────────────────────────────────────────────────┤
│  SlotResolver       "Ивану"→userId, "высокий"→PRIORITY, "готово"→       │  ТП-93 (F3)
│                     statusId, "эту"→открытая задача; неоднозначно →     │
│                     варианты для уточнения. НИКОГДА не выдумывает ID.    │
├──────────────────────────────────────────────────────────────────────┤
│  ConfirmationGate   riskLevel: safe → сразу · confirm → карточка ·      │  ТП-95 (X1)
│                     destructive → всегда явное подтверждение            │
├──────────────────────────────────────────────────────────────────────┤
│  CommandExecutor    registry.get(commandId).execute(slots, ctx)         │  ТП-91 (F1)
│                     → существующие мутации workTechApi/хуки             │
├──────────────────────────────────────────────────────────────────────┤
│  Feedback + Log     тост «Открыть/Отменить» (как useCreateTask);        │  ТП-95/103
│                     журнал действий + undo (позже — TTS)                │
└──────────────────────────────────────────────────────────────────────┘
```

**Принцип разделения:** LLM/резолвер **только выбирает команду и предлагает значения слотов**.
Приведение слотов к реальным сущностям — детерминированный `SlotResolver`. Исполнение —
детерминированный `CommandExecutor` через существующие мутации. Это анти‑галлюцинационный
барьер: модель не вызывает API и не изобретает ID.

## 4. Контракты (эскизы; не код — фиксируют форму для F1..X1)

```ts
// Поля-цель диктовки (сохраняем совместимость с ТП-88).
type VoiceField = 'title' | 'description' | 'comment'

// Уровень риска действия — задаёт поведение ConfirmationGate.
type RiskLevel = 'safe' | 'confirm' | 'destructive'

// Снимок контекста (F2). Только уже закэшированные данные.
type VoiceContext = {
  projectId: string
  activeSprintId?: string
  defaultSprintId: string
  openTask?: { id: string; code: string; title: string }
  currentUserId: string
  lookup: {
    members: Array<{ id: string; name: string; username?: string }>
    statuses: Array<{ id: number; code: string; label: string }>
    sprints: Array<{ id: string; name: string; active: boolean }>
    priorities: Array<{ value: string; label: string }>
  }
}

// Результат распознавания намерения (I1/I2). Единый DTO фронт↔бэк (OpenAPI).
type IntentResolution = {
  commandId: string | null      // null = не распознано
  slots: Record<string, string> // человекочитаемые значения ДО SlotResolver
  confidence: number            // 0..1
  needsConfirmation?: boolean    // подсказка резолвера сверх riskLevel команды
  clarification?: string         // уточняющий вопрос при неоднозначности
}

// Сервисы, инъецируемые командам (эволюция VoiceCommandContext из ТП-22).
type VoiceCommandContext = VoiceContext & {
  createTask: (dto: TaskModelDTO) => Promise<{ id: string; code: string }>
  updateTaskStatus: (taskId: string, statusId: number) => Promise<void>
  updateTasksSprint: (taskIds: string[], sprintId: string) => Promise<void>
  navigate: (to: string, params?: Record<string, string>) => void
  // …расширяется по мере добавления доменов (comments, sprints, meetings)
}

type VoiceCommandResult = { message: string; taskCode?: string; undo?: () => Promise<void> }

// Дескриптор команды (эволюция VoiceCommand из ТП-22: убран match()).
interface VoiceCommand<TSlots = Record<string, string>> {
  id: string
  description: string          // для схемы инструмента LLM
  examples: string[]           // few-shot фразы (и для RuleBased, и для LLM)
  riskLevel: RiskLevel
  slots: ZodSchema<TSlots>     // валидация + описание слотов
  resolve(raw: Record<string, string>, ctx: VoiceContext):
    | { ok: true; slots: TSlots }
    | { ok: false; clarification: string }   // неоднозначность/нехватка данных
  execute(slots: TSlots, ctx: VoiceCommandContext): Promise<VoiceCommandResult>
}

// Шов из ТП-88 сохраняется:
interface IntentAnalyzer { analyze(text: string, ctx: VoiceContext): Promise<IntentResolution> }
```

Диспетчеризация (эволюция `dispatchTranscript`):
`transcript → analyzer.analyze() → {commandId, slots} → gate(riskLevel) → command.resolve() →
command.execute() → feedback/log`. `UnrecognizedCommandError`/`clarification` сохраняются для
пути «не распознано / нужно уточнить».

## 5. Где живёт распознавание намерения (NLU)

> **Уточнение (повторный анализ ТП‑94):** правила НЕ требуют ключа, поэтому
> `RuleBasedResolver` работает **на клиенте** (без сетевого round‑trip) — Веха 1
> полностью офлайн. Backend‑прокси нужен ТОЛЬКО для LLM (там живёт секрет) и
> вводится в ТП‑96 (I2), а не в ТП‑94. Оба резолвера реализуют один фронт‑интерфейс
> `IntentResolver` (`resolve(text, ctx) → IntentResolution`), `RemoteResolver`
> делает авто‑фолбэк на `RuleBasedResolver`. Ниже описан именно LLM‑путь.

**Решение (LLM‑путь): backend‑прокси, провайдер за интерфейсом.** Фронт шлёт `POST /voice/resolve`
(`transcript` + `context` + схема команд из реестра) → бэкенд вызывает `IntentResolver`
(Strategy) → возвращает `IntentResolution`. Причины:
- **Секреты только на сервере** (SECURITY из CLAUDE.md; снимает блокер ТП‑10 без утечки ключа).
- **Провайдер сменяем через env** (как `app.dev-panel.github-repo`): RuleBased / OpenAI /
  Anthropic / YandexGPT / локальная модель.
- **Работа без AI и с AI за одним швом:** `RuleBasedResolver` даёт рабочий продукт без ключа;
  `LlmResolver` подключается позже; при отсутствии ключа/таймауте — авто‑откат на правила.

Источник истины о командах — **фронтенд‑реестр** (там же исполнение). Схема команд передаётся
прокси per‑request; бэкенд остаётся тонким stateless‑NLU и не знает про конкретные команды.
Безопасность сохраняется: исполняет клиент под JWT пользователя, каждая мутация проходит
серверные валидаторы/права.

## 6. Классификация риска и подтверждения

| Риск | Примеры | Поведение |
|---|---|---|
| **safe** | навигация, поиск, диктовка в поле, создание черновика | выполнить сразу, показать результат (+undo где есть) |
| **confirm** | смена статуса/исполнителя/приоритета/спринта, редактирование | карточка «что изменится» → одно подтверждение |
| **destructive** | удаление задачи/комментария, завершение спринта | всегда явное подтверждение + «зачитка» перед применением |

Обоснование: ASR может ошибиться в имени/коде задачи; подтверждение опасного и «read‑back»
(паттерн Linear/Jira «review before apply») предотвращают порчу данных. Соответствует
Rollback strategy из CLAUDE.md.

## 7. ADR‑записи

**ADR‑005. Голос — модальность ввода поверх существующих команд.**
Решение: голосовые действия выполняются только через существующие мутации/сервисы
(`TaskPlacementService` и т.д.). Причина: единый источник бизнес‑логики и контракта фронт↔бэк,
нет дублирования. Следствие: набор голосовых возможностей ограничен существующим API — новые
действия сперва появляются как обычные эндпоинты. Альтернатива (свой voice‑бэкенд с прямыми
операциями) отклонена — дублирование и рассинхрон инвариантов.

**ADR‑006. Реестр команд как механизм расширения (эволюция ТП‑22).**
Решение: каждая голосовая возможность = дескриптор `VoiceCommand` в реестре; добавление
команды не меняет существующие компоненты. Причина: масштабируемость («новые действия без
изменения компонентов»), проверено прежним кодом проекта. Следствие: `match()` заменён на
центральный `IntentResolver`; команда описывает себя (`description`, `examples`, `slots`) для
резолвера. Альтернатива (switch/if по интентам) отклонена — жёсткая связанность.

**ADR‑007. Распознавание намерения на бэкенд‑прокси, провайдер за интерфейсом.**
Решение: `POST /voice/resolve` + `IntentResolver` (Strategy), провайдер через env. Причина:
секреты только на сервере; работа с AI и без AI за одним швом; заменяемость провайдера.
Следствие: схема команд передаётся per‑request; нужен фолбэк RuleBased. Альтернатива (LLM с
фронта) отклонена — утечка ключа.

**ADR‑008. LLM выбирает и заполняет — система исполняет.**
Решение: резолвер возвращает `commandId`+слоты (человекочитаемые); `SlotResolver` детерминированно
приводит их к ID; `Executor` исполняет. Причина: анти‑галлюцинация (модель не трогает API/ID).
Следствие: нужен строгий `SlotResolver` и промпт «ничего не выдумывай». Альтернатива (LLM
формирует запрос к API) отклонена — риск выдуманных действий/ID.

**ADR‑009. ASR за интерфейсом `SpeechProvider`; дефолт — Web Speech (локально, без ключей).**
Решение: сохраняем `useSpeechRecognition` (ru‑RU, push‑to‑talk) как реализацию `SpeechProvider`.
Причина: локальная обработка без выгрузки аудио на наш бэкенд; заменяемость на облачный ASR.
Следствие: ограничения браузера (нет Firefox, зависимость от онлайн‑сервиса Chrome) → нужен
graceful‑fallback к вводу с клавиатуры и фиче‑детект. См. TD‑016.

**ADR‑010. Push‑to‑talk в MVP; полный дуплекс/barge‑in — отложены.**
Решение: MVP — «нажать микрофон → сказать → стоп» (надёжный базовый режим Wispr/Superwhisper).
Причина: full‑duplex (ChatGPT Realtime: server‑VAD, barge‑in, WebSocket) существенно сложнее и
не нужен для командной модели. Следствие: разговорный режим вынесен в пост‑MVP расширение
`SpeechProvider`, не в фундамент.
