# Итерационный Review для AI-разработчика — проектирование (ТП-157)

> Статус: ПРОЕКТ (код не пишется — решение ТП-157). Дата: 2026-07-06.
> Автор: Cloud Code. Владелец решения: владелец продукта.

## 1. Проблема

WorkTask становится средой, где реализацию выполняет AI (Cloud Code), а человек
принимает решения. Линейный цикл `To Do → In Progress → Review → Done` не
покрывает главный сценарий такой работы — **несколько попыток**: сейчас возврат
на доработку — это либо новая задача (теряется история, плодятся дубликаты),
либо переписка в комментариях без структуры (замечания, версии и результаты
перемешаны — реальный опыт ТП-118/ТП-39 этой доски).

Цель: **вся история разработки живёт внутри одной задачи** как последовательность
итераций `v1 → замечания → v2 → … → Одобрено → Done`.

## 2. Анализ существующих решений

| Система | Что берём | Что НЕ подходит и почему |
|---|---|---|
| **GitLab MR versions** | Ключевая идея: каждая волна пушей = «версия» диффа; комментарии привязаны к версии; сравнение любых двух версий | Версия = git-пуш; у нас итерация крупнее — «завершённая попытка» с тестами и скриншотами |
| **GitHub PR review rounds** | Цикл Request changes → повторный пуш → Re-review; статусы APPROVED / CHANGES_REQUESTED; «outdated» комментарии | Ревью размазано по диффу файлов — для нечитающего код владельца продукта неудобно как основной вид |
| **Figma version history + review** | Именованные версии с автором и описанием; «открыть любую прошлую версию» | Визуальный дифф картинок не наш случай (у нас — текстовый отчёт + скриншоты) |
| **Linear** | Минимализм карточки: активность одной лентой, статусы без церемоний | Итераций нет вовсе — возврат = переоткрытие без структуры |
| **Jira** | Ничего сверх комментариев/истории | Re-open теряет границы попыток; тяжёлые кастомные workflow — антипаттерн для нас |
| **YouTrack / Azure DevOps** | Azure: «Iterations» — только спринты (не то); YouTrack — workflow-скрипты | Терминология пересекается, механика не о том |
| **AI-IDE (Cursor checkpoints, Claude Code sessions)** | Понятие «checkpoint попытки», к которому можно вернуться; отчёт агента как артефакт | Локальны для сессии инструмента, не для трекера |

Вывод: наиболее близкая проверенная модель — **GitLab MR versions × GitHub review
rounds**, но поднятая с уровня «git-диффа» на уровень «попытка реализации задачи»
(отчёт + PR-ссылки + тесты + скриншоты), потому что ревьюер в WorkTask принимает
продукт, а не строки кода.

## 3. Концепция

**Итерация (Iteration)** — завершённая попытка реализации задачи. Создаётся в
момент перевода задачи в Review (автоматически из отчёта Cloud Code или вручную
человеком-разработчиком). Дальше только три исхода:

```
In Progress ──(готово)──▶ Review: создана vN
   ▲                          │
   │  CHANGES_REQUESTED       ├─ Одобрить ──▶ APPROVED → задача Done
   └──(замечания к vN)◀───────┤
                              └─ Комментарии к vN (не меняют статус)
```

- Возврат на доработку НЕ создаёт задачу и НЕ теряет историю: vN получает
  `CHANGES_REQUESTED` + замечания; следующая попытка = vN+1, vN становится
  `SUPERSEDED` при создании vN+1.
- Замечания адресуются конкретной версии — Cloud Code читает замечания
  «текущей» итерации, а не всю переписку.
- Статус задачи остаётся источником правды на доске (Review/In Progress/Done);
  итерации — структура ВНУТРИ задачи, доску не меняем (§2 бизнес-правил).

## 4. Модель данных (аддитивно, обратимо)

Новая таблица `task_iteration`:

| Поле | Тип | Смысл |
|---|---|---|
| id | uuid PK | |
| task_id | uuid FK → task_model (indexed) | Задача |
| number | int | v1, v2… (unique вместе с task_id) |
| status | varchar | IN_REVIEW / CHANGES_REQUESTED / APPROVED / SUPERSEDED |
| summary | varchar(255) | Однострочное «что сделано» |
| report | text | Markdown-отчёт: изменения, файлы, тесты, ограничения, известные проблемы |
| author_id | FK users | Кто создал (система/человек) |
| created_at / resolved_at | timestamptz | |

Связи с существующим (переиспользование, не дубли):
- **PR-ссылки** — уже есть dev-панель (GitHubDevPanelService, matching по коду
  задачи); итерация хранит только диапазон времени → PR фильтруются по нему.
  Опционально поле `pr_numbers varchar` для точной фиксации.
- **Комментарии** — `comment.iteration_id uuid NULL` (аддитивная колонка):
  комментарий из карточки в режиме просмотра vN привязывается к vN; старые
  комментарии (NULL) остаются общими для задачи.
- **Скриншоты «до/после»** — существующие вложения задачи; в отчёте (markdown)
  ссылки на attachmentId. Новых механизмов не заводим.
- **История задачи** (task_history) не трогается: итерации — своя лента.

Миграции: `CREATE TABLE task_iteration` + `ALTER TABLE comment ADD COLUMN
iteration_id uuid NULL` — обе аддитивные, откат = drop. Данные не мигрируются.

## 5. API (минимальный контракт)

- `GET  /tasks/{projectId}/{taskId}/iterations` — список (номер, статус, summary, created)
- `GET  /tasks/{projectId}/{taskId}/iterations/{n}` — отчёт + комментарии версии
- `POST /tasks/{projectId}/{taskId}/iterations` — создать vN (summary, report);
  side-effect: предыдущая IN_REVIEW/CHANGES_REQUESTED → SUPERSEDED; задача → Review
- `PUT  /tasks/{projectId}/{taskId}/iterations/{n}/approve` — APPROVED; задача → Done
- `PUT  /tasks/{projectId}/{taskId}/iterations/{n}/request-changes` — CHANGES_REQUESTED;
  задача → In Progress (тело: markdown замечаний → комментарий, привязанный к vN)

Комментарии версии — существующий `create-comment` + `iterationId`.
Права: как у комментариев/статусов (участник проекта). Всё через
TaskPlacementService-инварианты не проходит — статусы меняются существующим
updateTaskStatus (единая точка сохраняется).

## 6. UX (не перегружая карточку)

Карточка уже имеет вкладки «Комментарии / Связи / История» — итерации становятся
**первой вкладкой «Итерации»** (появляется только если у задачи ≥1 итерация;
задачи без AI-цикла выглядят как раньше — ноль визуального шума).

Содержимое вкладки (паттерн GitLab versions + Linear activity):
- **Шапка: последняя итерация** — крупно: `v3 · На проверке · 2 ч назад`,
  summary, кнопки владельца: `Одобрить` (primary) / `Вернуть на доработку`
  (secondary, открывает поле замечаний — обязательное).
- **Отчёт** — markdown-рендер (что сделано/файлы/тесты/скриншоты/ограничения);
  PR-чипы из dev-панели за период итерации.
- **Лента версий** — свёрнутый список v2, v1 (статус-чип: серый SUPERSEDED,
  оранжевый CHANGES_REQUESTED, зелёный APPROVED); клик — открыть версию целиком
  (отчёт + её замечания). «Сравнить с предыдущей» — этап 2 (текстовый дифф отчётов).
- **Комментарии в контексте версии**: composer внизу вкладки пишет в текущую
  просматриваемую версию (подпись «к v2»); общая вкладка «Комментарии» остаётся.

Статус-чипы, кнопки, markdown — существующие компоненты дизайн-системы
(Chip, ConfirmDialog для «Вернуть», TaskDescriptionField-паттерн для замечаний).

Пользовательский сценарий (главный):
1. Cloud Code завершил работу → задача в Review, на карточке вкладка «Итерации», v1.
2. Владелец читает summary/отчёт/скриншоты, жмёт «Вернуть на доработку», пишет замечания.
3. Задача в In Progress; Cloud Code берёт замечания v1 → делает → создаёт v2.
4. Владелец: «Одобрить» → v2 APPROVED, задача Done. Вся история — в одной карточке.

## 7. Интеграция с Cloud Code (потребитель №1)

Сегодняшний комментарий-отчёт «[Cloud Code] Выполнено, PR #N…» — это неструктурированная
итерация. После этапа 1 бот вместо комментария вызывает `POST /iterations`
(summary + тот же markdown). Возврат из Review читается как `CHANGES_REQUESTED`
+ замечания версии — точный контекст без парсинга всей переписки. Механизм
одинаково работает и для человека-разработчика (создать итерацию вручную из карточки).

## 8. Риски

| Риск | Митигция |
|---|---|
| Перегрузка карточки | Вкладка появляется только при наличии итераций; версии свёрнуты |
| Двойная правда «комментарии vs замечания версии» | Замечания = те же комментарии с iteration_id; одна сущность, два среза |
| Расхождение статуса задачи и итерации | Статусы итераций меняются ТОЛЬКО через approve/request-changes/create, которые сами двигают задачу (транзакционно) |
| Терминология («итерация» ≠ спринт) | В UI слово «Версия» (v1, v2) — как в GitLab/Figma; в коде Iteration |
| Захламление таблицы | Итерации — по явному действию, не на каждый комментарий |

## 9. План реализации по этапам (каждый — самостоятельная ценность)

1. **Ядро**: миграции, entity, API list/create/approve/request-changes,
   автопереходы статусов задачи; unit-тесты. (backend, ~1 задача)
2. **Карточка**: вкладка «Итерации» — последняя версия + лента + действия
   владельца. (frontend, ~1-2 задачи)
3. **Комментарии версии**: iteration_id в composer + отображение в версии. (1 задача)
4. **Интеграция Cloud Code**: бот создаёт итерации вместо комментариев-отчётов;
   регламент чтения замечаний. (процесс + скрипт)
5. Пост-MVP: сравнение версий, счётчик итераций на карточке доски, уведомления
   «возвращено на доработку»/«одобрено».
