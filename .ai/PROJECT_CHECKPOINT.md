## Завершено 2026-07-04: ТП-81 «Невозможно редактировать спринт» BUG (ветка `fix/tp81-sprint-name-nullable`)

При очистке названия спринта сохранение падало с 400 «нарушено ограничение
целостности данных». Причина: ТП-70 сделал имя спринта опциональным (DTO без
@NotNull, `normalizeName('')→null`), но **колонка `sprint.name` в БД осталась
NOT NULL** из исходной миграции — предположение ТП-70 «колонка уже nullable»
было неверным. Пустое имя → null → нарушение NOT NULL.
- **Фикс:** миграция `20260709-sprint-name-nullable.xml` —
  `dropNotNullConstraint` на `sprint.name`. БД приведена в соответствие с моделью
  (`@Column`, nullable) и намерением ТП-70. Java-код не менялся.
- **Диагностика:** воспроизведено на локальном backend из main против общей БД
  (curl-тесты с корректным UTF-8 через node): rename→200, **clear name→400**,
  no-dates→200. После применения миграции (рестарт, Liquibase: «Null constraint
  dropped from sprint.name») clear name→200 (name=null). Урок: локальный backend
  на :8080 из другой сессии был на СТАРОМ коде (хранил пустое имя как ""), поэтому
  бага там не воспроизводилась — проверять на актуальном main (branch/merge integrity).
- Тесты: полный backend `./gradlew test` зелёный (юнит-тесты мокируют репозиторий,
  ограничение БД проверяется интеграционно на реальной БД — сделано вручную).
