# MEET_ARCHITECTURE.md — Подсистема онлайн-встреч WorkTask Meet

> ADR и проектный документ подсистемы видеовстреч. Статус: принят (гейт пройден до кода).
> Заголовочные решения дублируются в `.ai/ARCHITECTURE.md` (ADR-011..ADR-016).

---

## 1. Продуктовая рамка

**Проблема.** Встречи в WorkTask сейчас — записи календаря с полем «ссылка», ведущим во
внешние сервисы (Телемост и т.п.). Команда выпадает из продукта: нет связи звонка с
проектом/задачей, нет прав, нет уведомления «встреча началась», голосовой помощник может
создать встречу, но не может её «провести».

**Решение.** WorkTask Meet — встроенные видеовстречи: комната с неугадываемой ссылкой
`/meet/{token}`, создаваемая из календаря, задачи или мгновенно; вход в один клик для
участников проекта; аудио/видео/демонстрация экрана/чат; уведомления и голосовое управление
поверх существующих механизмов.

Meet **дополняет**, а не заменяет внешние ссылки: поле `Meeting.link` продолжает принимать
любой URL; Meet просто умеет заполнять его собственной ссылкой.

## 2. Запись решения (операционный цикл)

1. **Решение.** WebRTC full-mesh (P2P) для комнат до 8 участников; сигналинг — новый
   WebSocket-эндпоинт в существующем Spring Boot backend; состояние комнат — in-memory
   реестр; комната привязана к проекту (права = членство в проекте); токен комнаты —
   `SecureRandom` 32 байта base64url; ICE-конфигурация отдаётся backend'ом (STUN по
   умолчанию, TURN подключается переменными окружения без изменения кода).
2. **Альтернативы.** (а) SFU self-hosted (LiveKit/mediasoup/Janus/Jitsi VB); (б) managed
   SFU (LiveKit Cloud, Twilio); (в) сигналинг поверх существующего поллинга; (г) STOMP/SockJS;
   (д) оставить только внешние ссылки.
3. **Почему mesh + свой сигналинг.** (а)/(б) — отдельный медиасервер или платный сервис:
   инфраструктурные затраты и лицензионные решения — граница владельца продукта
   (см. §9); mesh покрывает реальный сценарий команды (проект ~2–8 человек) без новой
   инфраструктуры, с медиашифрованием DTLS-SRTP по умолчанию. (в) — поллинг 10–30с
   непригоден для обмена offer/ICE (нужны сотни мс). (г) — STOMP тянет брокерную
   семантику ради одного канала; протокол комнаты (host-контролы, präсенс) всё равно
   кастомный; raw WebSocket + JSON-конверт проще и полностью тестируем. (д) — не решает
   продуктовую проблему.
4. **Blast radius.** Новая таблица `meet_room`; `Notification` — новый тип (аддитивно);
   `SecurityConfig` — разрешение handshake-пути WS (точечно); nginx — upgrade-заголовки
   для `/work-task/ws/` (аддитивный location); календарь/карточка встречи/карточка задачи —
   новые кнопки; голосовой реестр — новые команды. Существующие контракты не меняются.
5. **Обратимость.** Каждый этап — отдельный PR с revert без ручного восстановления:
   новая таблица, новые классы, новые компоненты; правки существующих файлов — точечные
   добавления. Отключение фичи = revert PR фронтенда (backend-эндпоинты пассивны).

## 3. Топология медиа (ADR-011)

**Full-mesh P2P, жёсткий предел 8 участников.**

- Каждая пара участников соединена собственным `RTCPeerConnection` (DTLS-SRTP из коробки).
- N участников: исходящих потоков на клиента — (N−1). До 4 участников — полноценное видео;
  5–8 — приемлемо на офисных каналах (адаптация битрейта решает); >8 — вход блокируется
  сообщением «Встреча заполнена» ещё на сервере (`ROOM_FULL`).
- Ограничения качества: video 720p max, `degradationPreference: 'maintain-framerate'` не
  задаём — приоритет отдаём звуку (см. §7).
- **Порог масштабирования** зафиксирован константой `MAX_PARTICIPANTS = 8` на сервере
  (единственный источник) и транслируется клиенту в состоянии комнаты.

**Путь масштабирования (решение владельца, см. §9):** при потребности в комнатах >8 —
зрелый SFU (кандидат №1 LiveKit self-hosted: активная поддержка, Apache-2.0, готовые
SDK; альтернатива — Jitsi Videobridge). Архитектура к этому готова: сигналинг и модель
комнаты не зависят от топологии, меняется только фабрика соединений на клиенте
(`PeerConnectionFabric`) и появляется медиасервер в инфраструктуре. Собственный SFU не
пишем ни при каком сценарии.

## 4. Сигналинг (ADR-012)

**Raw WebSocket `wss://…/work-task/ws/meet?token={JWT}&room={roomToken}` в том же
Spring Boot приложении.**

- Существующая realtime-«инфраструктура» продукта — короткий поллинг TanStack Query
  (10–30с) — по латентности непригодна для SDP/ICE. `.ai/ARCHITECTURE.md` числит
  «WebSocket/SSE» нерешённым вопросом, а `pollingConfig.ts` прямо закладывает переход на
  WebSocket отдельным шагом. Meet **закрывает этот вопрос**: вводимый WS-слой — общий
  (`ru.worktechlab.work_task.ws`), сигналинг Meet — первый потребитель; миграция доски/
  уведомлений на него — отдельные будущие задачи (без обязательств в этом эпике).
- Аутентификация handshake: JWT access-token в query-параметре (браузерный WebSocket не
  умеет заголовки); валидация тем же `JwtUtils`, состав пользователя — тем же
  `UserDetailsService`. Путь `/work-task/ws/**` исключается из цепочки HTTP-фильтра
  (STATELESS, токен обязателен на handshake — эквивалентная защита).
- Протокол: JSON-конверт `{type, ...payload}`. Типы: `hello` (сервер → снапшот комнаты),
  `peer-joined/peer-left`, `offer/answer/ice` (адресная переправка, сервер не читает SDP),
  `chat`, `state` (mute/camera/screen — присенс), `host-mute`, `host-remove`, `room-closed`,
  `error`, `ping/pong` (25с, таймаут 60с).
- Правило установления пары (perfect negotiation, «вежливость» по порядку входа):
  **новичок инициирует offer каждому существующему участнику**; при glare — вежливая
  сторона откатывается (`polite = joinedAt старше`).

## 5. Состояние встречи (ADR-013)

**Источник истины — backend, in-memory `MeetRoomRegistry` (ConcurrentHashMap).**

- Постоянное (`meet_room`, PostgreSQL): id, token (unique), project_id, meeting_id?,
  task_id?, title, created_by, created_at, ended_at?. Комната — долгоживущая ссылка:
  «встреча» может собираться по ней многократно (как ссылка Google Meet).
- Эфемерное (registry): подключённые участники (userId, sessionId, displayName, mute/cam/
  screen-флаги, joinedAt), баны текущей сессии (после host-remove). Приложение —
  единственная нода (как и весь WorkTask); при рестарте backend'а клиенты переподключаются
  автоматически (см. §7) — комната собирается заново из тех же людей.
- Синхронизация: каждый вход/выход/смена состояния рассылается всем в комнате
  (`peer-joined`/`peer-left`/`state`); полный снапшот — в `hello` при (пере)подключении.

## 6. Права и безопасность встречи (ADR-014)

- **Ссылка** — токен 32 случайных байта (base64url, 43 символа), генератор —
  существующий `JwtUtils.generateSecureToken`. Никаких последовательных ID в URL.
- **Авторизация входа**: WS-handshake требует валидный JWT; резолв комнаты по токену
  дополнительно проверяет **членство в проекте комнаты** тем же механизмом, что весь
  продукт (`CheckerUtil.findAndCheckProjectUserData`). Не участник проекта — `403` даже
  со ссылкой. Гостевой доступ по ссылке без аккаунта — **осознанно не реализуется** в
  этом эпике (продуктовая развилка про внешних участников — владельцу, §9).
- **Host** = создатель комнаты; если его нет в комнате — самый ранний вошедший (переход
  роли рассылается в `state`). Контролы: `host-mute` (принудительный self-mute на
  клиенте цели + рассылка), `host-remove` (сервер закрывает сессию цели, банит userId до
  конца сессии комнаты, рассылает `peer-left reason=removed`).
- **Медиа**: DTLS-SRTP — обязательное свойство WebRTC, ключи договариваются в DTLS,
  сервер медиапоток не видит вовсе (P2P). E2EE сверх этого — вне рамок (граница §9).
- **Демонстрация экрана** — только явное действие пользователя (`getDisplayMedia`
  невозможен без жеста и выбора источника браузером). Запись встреч — не реализуется
  (граница §9).
- Rate-limit сигналинга: сервер молча отбрасывает >40 сообщений/с с сессии (защита от
  флуда чата/ICE-штормов).

## 7. Качество соединения (ADR-015)

- **Переподключение**: WS — экспоненциальный backoff (0.5с → 8с, jitter), состояние
  `reconnecting` в UI; ICE — `restartIce()` при `iceConnectionState: failed`, пересоздание
  RTCPeerConnection при `connectionState: failed` после restartIce-попытки. Возврат в
  комнату после обрыва — тот же `join`, сервер выдаёт свежий снапшот.
- **Адаптация к каналу**: штатный congestion control WebRTC (transport-cc/REMB — браузер);
  явные потолки: видео ≤ 960×540@24 при ≥5 участниках (mesh-экономия), screen-share
  `contentHint='detail'`.
- **Приоритет звука**: аудио-трек всегда добавляется первым; при деградации канала браузер
  жертвует видео (аудио имеет приоритет по умолчанию в bandwidth-аллокации); UI позволяет
  выключить входящее видео («аудио-режим»).
- **NAT/TURN-готовность**: список ICE-серверов клиент получает с backend
  (`GET /meet/ice-servers`): по умолчанию `stun:stun.l.google.com:19302` (+резерв);
  TURN добавляется env-переменными `MEET_TURN_URLS/USERNAME/CREDENTIAL` — без изменения
  кода. Хранение TURN-креденшелов — секреты, выдаёт владелец (§9).
- **Наблюдаемость**: клиент пишет итог попытки соединения (успех/ICE-fail/время установки,
  число реконнектов) в `POST /meet/rooms/{token}/stats` — агрегат в логах backend
  (`MeetStatsService`), без внешних систем. Это даёт «долю успешных установок» из
  требований без новой инфраструктуры.
- **Отказы устройств**: страница работает без камеры/микрофона (recvonly + плитка с
  инициалами); отказ в доступе — объяснение и кнопка «Повторить»; неподдерживаемый
  браузер (нет `RTCPeerConnection`/`getUserMedia`) — понятный экран-заглушка с
  рекомендацией браузера, не белый экран.

## 8. UX-принципы

- Лобби перед входом (как Meet/Zoom): превью камеры, выбор устройств, состояние
  разрешений, кнопка «Присоединиться» — и только после неё медиа уходит другим.
- Единая дизайн-система: MUI v7 + токены `--wt-*` (тёмная/светлая), NiceModal для
  диалогов, нотификации через `notify`, размеры/отступы как в остальном продукте.
- Доступность: все контролы — кнопки с `aria-label`/`aria-pressed`, tooltips, клавиатурная
  навигация, список участников — семантический список; видимый фокус (стандарт MUI).
- Адаптив: сетка видео 1–8 плиток (auto-fit), мобильный — колонка + нижняя панель
  контролов; панели «Участники»/«Чат» — Drawer на мобильном, боковая панель на десктопе.

## 9. Границы — решения владельца продукта (STOP-список эпика)

Зафиксированы, работа по ним НЕ ведётся до явного решения:

1. **SFU / масштаб >8 участников** — выбор self-hosted vs managed, затраты на
   медиасервер/трафик.
2. **TURN-сервер** — нужен для «жёстких» NAT (симметричные, часть корпоративных сетей);
   креденшелы/hosted-провайдер — секреты и затраты. Без TURN P2P устанавливается в
   типичных сетях через STUN; при недоступности пары UI честно сообщает об ошибке
   соединения.
3. **Запись встреч / транскрипция / хранение медиа** — согласие участников, приватность,
   data residency.
4. **E2EE сверх DTLS-SRTP** (insertable streams).
5. **Гостевой доступ без аккаунта** (внешние участники) — модель прав/приглашений.

## 12. Исследованные референсы

Google Meet / Zoom / Teams / Slack Huddles / Discord / Телемост: (1) все групповые —
SFU/MCU на своей инфраструктуре; huddles/малые звонки исторически начинались с mesh
(Slack Huddles ~до 4–8, Discord DM); (2) стандартный UX-канон: лобби с превью → сетка
плиток → нижняя панель (mute/cam/share/leave) → боковые панели участников/чата — Meet
следует ему; (3) ссылка-код без логики в URL; (4) host-контролы: mute participant,
remove, ограничение share. Спецификации: WebRTC 1.0 (W3C), ICE RFC 8445, TURN RFC 8656,
DTLS-SRTP RFC 5763/5764, perfect negotiation (MDN/W3C pattern).
